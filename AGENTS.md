# Правила работы ИИ-агента с проектом yashil-energiya

Репозиторий: `github.com/islomalarov/yashil-energiya` · Прод: `yashil-energiya.uz` · Деплой: Vercel

> Главный принцип: **сначала изучи существующее решение — потом меняй**. Проект уже имеет настроенные зависимости, маршрутизацию, i18n, слой данных и конвенции. Любое изменение обязано встраиваться в них, а не создавать параллельные механизмы.

---

## 1. Технологический стек (зафиксирован)

| Слой | Технология / версия | Примечание |
|---|---|---|
| Runtime | Node.js **20.x** (`engines`, `.nvmrc`), пакетный менеджер — **npm** (`package-lock.json`) | не использовать yarn/pnpm/bun |
| Фреймворк | **Next.js ^16** (App Router), **React 19**, **TypeScript ^6** | dev запускается с `NEXT_TURBOPACK=0` — не включать Turbopack самовольно |
| Стили | **SCSS-модули** (`sass`) + `classnames` | не добавлять Tailwind, styled-components, CSS-in-JS |
| i18n | **next-intl ^4**, локали `en / ru / uz` в `src/i18n/routing.ts`, словари в `messages/` | вся маршрутизация локализована |
| CMS | **Hygraph**, клиент `graphql-request` + `graphql`, эндпоинт `NEXT_PUBLIC_HYGRAPH_ENDPOINT` | контент и медиа (`us-west-2.graphassets.com`) |
| Формы / почта | **Microsoft Graph** через `@azure/msal-node` (`MS_*` переменные), серверная валидация env в `src/lib/server-env.ts` | также в зависимостях есть `resend` — уточнить актуальный канал перед правками |
| Антибот | **Cloudflare Turnstile** (`@marsidev/react-turnstile`, `TURNSTILE_SECRET_KEY`) | не удалять и не обходить в формах |
| Rate limiting | **Upstash Redis** (`@upstash/ratelimit`, `@upstash/redis`) | защита API-роутов |
| Карты | **Leaflet + react-leaflet + leaflet.markercluster** | карта зарядных станций; рендерить только на клиенте (dynamic import, ssr:false) |
| Карусели | **Embla** (`embla-carousel`, `-react`, `-autoplay`) | не подключать swiper/slick |
| Графики | **Recharts** | |
| Иконки | **lucide-react** (+ `@iconify/react` в dev) | не добавлять другие icon-паки |
| Лайтбокс | **yet-another-react-lightbox** | |
| Уведомления | **react-toastify** | |
| Изображения | `next/image` + `sharp`; OG-картинки — API-роут `/api/og-image` (использует `sharp`/`text-to-svg`) | внешние домены картинок — только через `images` в `next.config.js` |
| Аналитика | `@vercel/analytics`, `@vercel/speed-insights`, GA (`NEXT_PUBLIC_GA_MEASUREMENT_ID`) | |
| Качество кода | ESLint 9 (`eslint.config.mjs`), Prettier (`.prettierrc`), `tsc` (`tsconfig.typecheck.json`) | |

**Запрещено:** менять стек, добавлять дублирующие библиотеки (второй UI-кит, второй слайдер, второй способ стилизации, axios вместо graphql-request/fetch), включать/обновлять мажорные версии Next/React в рамках фич-задач.

## 2. Структура репозитория (соблюдать при размещении кода)

```
.github/     — CI/CD и конфигурация GitHub
data/        — статические данные
docs/        — документация проекта (читать перед крупными задачами)
hooks/       — переиспользуемые React-хуки
interface/   — TypeScript-типы и интерфейсы
lib/         — утилиты
messages/    — словари переводов next-intl (en, ru, uz)
public/      — статика (логотипы, socialsLogo/, hero и т.д.)
scripts/     — служебные скрипты
services/    — слой работы с данными (GraphQL-запросы к Hygraph и пр.)
src/         — приложение (App Router, i18n, server-env и т.д.)
```

Правила:
1. Новый код кладётся в существующую папку по назначению: типы → `interface/`, хук → `hooks/`, запрос к CMS → `services/`, утилита → `lib/`. Новые «корневые» папки не создаются без согласования.
2. Перед задачей прочитать `README.md`, `docs/`, `REDESIGN_NOTES.md` и `TZ-redesign-chargingstation.md` — там зафиксированы принятые решения и действующее ТЗ по разделу зарядных станций.
3. Соблюдать стиль Prettier/ESLint проекта; не переформатировать чужие файлы «заодно».

## 3. Маршрутизация и i18n

1. Все страницы живут под сегментом локали (`/en`, `/ru`, `/uz`); маршрутизация управляется **next-intl** (`src/i18n/routing.ts` + middleware). Запрещено создавать страницы вне этой схемы или обходить middleware.
2. Карта публичных маршрутов (не ломать URL — на них завязаны SEO, canonical и внешние ссылки):
   - `/{locale}` — главная; `/{locale}/about`; `/{locale}/documents`; `/{locale}/contacts`
   - `/{locale}/plants` (пагинация `?page=N`) и `/{locale}/plants/[id]` — id записи Hygraph
   - `/{locale}/news` и `/{locale}/news/[slug]` — человекочитаемый slug из CMS
   - `/{locale}/installation-request`; `/{locale}/chargingstation/...` (в т.ч. `public-offer`)
   - `/api/og-image` и прочие route handlers
3. Новая страница = страница на **всех трёх языках**: добавить ключи во все файлы `messages/*`, метаданные через существующий механизм `generateMetadata` (title-шаблон `… | Yashil Energiya`, description, canonical, OG через `/api/og-image`).
4. Новые списки используют существующий паттерн пагинации `?page=N` и общий компонент пагинации.
5. Ссылки между страницами — через локализованные навигационные хелперы next-intl (`Link`/`useRouter` из `src/i18n/…`), а не «голый» `next/link` с ручной подстановкой локали.
6. Изменение схемы URL или удаление страниц — только по явному согласованию и с 301-редиректами.

---

## 4. Данные и интеграции

1. **Hygraph — единственный источник контента** (новости, СЭС, документы, зарядные станции). Контент не хардкодится. Запросы — в `services/` через `graphql-request`; перед написанием нового запроса найти и переиспользовать существующие запросы/фрагменты и типы из `interface/`.
2. Запросы выполняются с учётом текущей локали — следовать образцу существующих запросов. Стратегию кэширования/ревалидации (ISR) копировать с соседних страниц того же раздела.
3. **Формы обратной связи** — не трогать без необходимости связку: клиент → route handler → валидация env (`src/lib/server-env.ts`) → Turnstile-проверка → rate limit (Upstash) → отправка через Microsoft Graph. При изменении форм все звенья цепочки сохраняются.
4. **Секреты** — только через переменные окружения из `.env.example` (`NEXT_PUBLIC_HYGRAPH_ENDPOINT`, `TURNSTILE_*`, `MS_*`, `UPSTASH_*`, `NEXT_PUBLIC_GA_MEASUREMENT_ID`). Новые секреты добавляются в `.env.example` (без значений) и в валидацию `server-env.ts`. Реальные значения никогда не коммитятся.
5. Изображения из CMS и статики — только через `next/image`; новые внешние хосты добавлять в `next.config.js`, а не использовать `<img>`.
6. Leaflet-компоненты — только client-side (`'use client'` + dynamic import без SSR), иначе билд упадёт.

---

## 5. Компоненты и UI

1. **Переиспользование прежде создания.** Перед новым компонентом искать существующий: карточки (СЭС/новости), пагинация, хлебные крошки, hero-блок, шапка с выпадающими меню, футер, переключатель языков, поиск, тосты, лайтбокс, карусель Embla.
2. Новые страницы подключаются к общему layout (шапка/футер), не дублируют его.
3. Стили — SCSS-модуль рядом с компонентом, условные классы через `classnames`, повторное использование существующих переменных/миксинов SCSS.
4. Все пользовательские строки — через `next-intl` (`useTranslations`/`getTranslations`) с ключами во всех трёх словарях. Захардкоженная строка на одном языке = незавершённая задача.
5. Проверять адаптивность: мобильное бургер-меню и десктоп.

---

## 6. Зависимости

1. Новая зависимость — только если в проекте нет решения, её нельзя заменить небольшим собственным кодом, и она поддерживается. Обоснование — в описании PR.
2. `npm install` → коммитить `package.json` **вместе с** `package-lock.json`. Не пересоздавать lock-файл другим менеджером.
3. Мажорные обновления (Next, React, ESLint, TypeScript и пр.) — отдельная задача в отдельной ветке `chore/…`, никогда «заодно» с фичей.

---

## 7. Git-процесс

1. **`main` защищена — прямые коммиты запрещены.** `main` = то, что деплоится на Vercel.
2. **Любая крупная доработка выполняется в отдельной ветке**: новый раздел, изменение маршрутизации/i18n, правки моделей CMS/запросов, редизайн (см. `TZ-redesign-chargingstation.md`), обновление зависимостей, рефакторинг.
   - `feature/<короткое-описание>` — функциональность
   - `fix/<короткое-описание>` — исправления
   - `chore/<короткое-описание>` — зависимости/конфиги/инфраструктура
3. Одна ветка = одна задача. Не смешивать рефакторинг, форматирование и фичи в одном PR.
4. Коммиты атомарные, сообщения осмысленные (что и зачем). Перед PR ветка актуализирована от `main`.
5. PR обязан содержать: список изменений, затронутые маршруты и локали, новые env-переменные (если есть), скриншоты UI-изменений. Мерж — только после ревью; агент не мержит сам.
6. Vercel строит preview-деплой для веток — проверять изменения на preview-URL до мержа.

## 8. Обязательные проверки перед сдачей (Definition of Done)

Выполнить локально, всё должно проходить без ошибок:

```
npm run lint
npm run typecheck
npm run build
```

Дополнительно:
1. Изменённые страницы проверены на **всех трёх локалях** (`/en`, `/ru`, `/uz`).
2. Не сломаны: шапка/меню, футер, переключатель языков, поиск, пагинация, формы (включая Turnstile), карта, метатеги изменённых страниц.
3. Нет ошибок `next/image` («hostname is not configured»), нет SSR-ошибок Leaflet.
4. В диффе нет секретов, `.env`, мусорных файлов и массового переформатирования.

---

## 9. Запрещено без явного согласования владельца

- Менять схему URL, canonical, удалять опубликованные страницы.
- Менять модели/контент в Hygraph.
- Отключать или ослаблять Turnstile и rate limiting.
- Менять конфигурацию отправки почты (Microsoft Graph) и получателей.
- Коммитить секреты и реальные `.env`.
- Мажорные обновления и «большой рефакторинг заодно».
- Мержить в `main` без ревью.

---

## 10. Алгоритм работы агента над задачей

1. Прочитать задачу → определить затронутые маршруты, компоненты, `services/`-запросы, словари `messages/` и env.
2. Изучить существующий код этих участков и документацию (`docs/`, `REDESIGN_NOTES.md`, ТЗ).
3. Создать ветку по разделу 7.
4. Реализовать изменение, максимально переиспользуя существующие компоненты, хуки, типы и запросы.
5. Добавить переводы en/ru/uz и метаданные.
6. Прогнать `lint` / `typecheck` / `build` и проверки раздела 8.
7. Оформить PR, проверить preview-деплой Vercel, дождаться ревью. **Не мержить самостоятельно.**
