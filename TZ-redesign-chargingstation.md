# ТЗ: Редизайн страницы «Электрозарядные станции» (/[locale]/chargingstation)

Проект: yashil-energiya.uz · Репозиторий: `C:\yashil-energiya-new`

---

## 0. Контекст проекта (что известно заранее)

- Сайт на **Next.js** (image optimization через `next/image`, пути `/_next/*`).
- Headless CMS — **Hygraph (GraphCMS)**, ассеты с `us-west-2.graphassets.com`. Новости и СЭС (plants) — CMS-driven.
- **Локализация на 3 языка**: маршруты `/en`, `/ru`, `/uz`.
- Есть кастомный роут `/api/og-image` для OG-изображений.
- Общий шаблон внутренних страниц: hero-баннер с заголовком и хлебными крошками («Home page | Название») → контент → блок «Контакты» (телефон, адрес, ориентир UZGIDROMET, e-mail, соцсети) → футер со ссылкой на публичную оферту ЭЗС.
- Фирменная тема — «зелёная энергетика»: солнце, гидро, EV. Визуальный язык: карточки с фото и характеристикой (как у СЭС: «1500 kW» бейдж + фото + название + локация + кнопка Detailed), вертикальный таймлайн на главной.
- Ключевые факты из контента сайта, которые можно использовать: ЭЗС запущены в **Кашкадарьинской, Джизакской и Бухарской областях** (2026); компания — СП с эмиратской GSU (51%); установленная мощность активов 130 000 кВт (2025).

## 1. Этап разведки (обязателен, выполнить ДО написания кода)

1. Изучить репозиторий: `package.json` (версия Next, роутер App/Pages, стилизация — Tailwind / CSS Modules / SCSS / styled-components, i18n-библиотека — next-intl / next-i18next / кастом).
2. Найти текущую страницу chargingstation (`app/[locale]/chargingstation/...` или `pages/...`) и страницу `chargingstation/public-offer`.
3. Составить инвентарь переиспользуемых компонентов: Hero/PageBanner с хлебными крошками, карточка (как у plants), таймлайн, аккордеон (если есть), кнопки, секция контактов, контейнер/сетка, типографика.
4. Найти дизайн-токены: цвета бренда (основной зелёный, акценты), шрифты, радиусы, тени, брейкпоинты. **Новые стили строить только на существующих токенах** — никаких новых цветов «от себя».
5. Проверить, как хранится контент: если текст текущей страницы лежит в Hygraph — посмотреть модель и запросы (GraphQL); если в локальных словарях (`messages/en.json` и т.п.) — работать с ними. Выяснить, есть ли в CMS модель для ЭЗС (аналог модели plants). Если есть — использовать её как источник данных для списка станций.
6. Проверить, как формируются metadata/SEO на других страницах, и повторить паттерн.
7. Кратко зафиксировать результаты разведки в комментарии/файле `REDESIGN_NOTES.md` перед началом правок.

## 2. Цели редизайна

- Превратить «стену текста» в информативную продуктовую страницу уровня остального сайта.
- Показать реальную сеть станций (регионы, характеристики), а не только рассуждения о пользе EV.
- Дать пользователю практическую информацию: типы зарядки, коннекторы, как воспользоваться, тарифы, оферта.
- Сохранить полную визуальную преемственность: тот же header/footer, hero-паттерн с хлебными крошками, карточный стиль, фирменные цвета и типографика.
- Полная поддержка EN/RU/UZ с первого коммита.

## 3. Новая структура страницы (сверху вниз)

### 3.1. Hero
- Существующий hero-паттерн (баннер + заголовок + хлебные крошки), но с EV-тематическим изображением (станция/электромобиль). Если подходящего ассета нет — использовать текущий hero и пометить `TODO: заменить изображение`.
- Подзаголовок в одну строку: миссия направления (развитие зарядной инфраструктуры для чистого транспорта в Узбекистане).

### 3.2. Полоса ключевых показателей (stats strip)
- 3–4 метрики в ряд: количество станций / количество коннекторов / охваченные регионы / суммарная мощность.
- Данные брать из CMS, если модель существует; иначе — вынести в словари локализации и проставить `TODO: уточнить у заказчика реальные цифры`. **Не выдумывать конкретные числа.** Можно безопасно указать «3+ региона» (Кашкадарья, Джизак, Бухара — подтверждено сайтом).
- Анимация count-up при появлении в viewport — только если на сайте уже есть анимации (framer-motion и т.п.); не тащить новую зависимость ради этого.

### 3.3. «Почему это важно» — редизайн текущего текста
- Существующие блоки («Экология», «Эффективное использование энергии», «Снижение транспортных расходов», «Устойчивое энергоснабжение», «Экономический рост», «Удобство», «Цифровая интеграция») переработать в **сетку иконка-карточек** (2×3 или 3×2, на мобильном — колонка).
- Каждая карточка: иконка (SVG в стиле сайта, монохромный зелёный), короткий заголовок, 1–2 предложения. Тексты сократить и отредактировать — сейчас английская версия местами машинная («the air harmful gases», «recharge stations»); переписать грамотно на всех трёх языках.
- Раздел «Статистика и прогнозы» либо сжать до 2–3 фактов в этой же секции, либо интегрировать в stats strip.

### 3.4. Сеть станций
- Секция «Наши зарядные станции» / «Карта сети».
- **Вариант A (если в CMS есть данные по станциям):** карточки станций в стиле карточек СЭС — фото, бейдж мощности (кВт), название/локация, кол-во коннекторов, типы (AC/DC), кнопка Detailed при наличии детальных страниц.
- **Вариант B (данных нет):** стилизованная карта Узбекистана (SVG) с подсвеченными регионами присутствия (Кашкадарья, Джизак, Бухара + Ташкент, если подтвердится по данным репо/CMS) и легендой. Не использовать тяжёлые map-библиотеки (Leaflet/Mapbox), если их нет в проекте.
- Заложить структуру данных так, чтобы позже переключиться с локального массива на CMS без переписывания UI.

### 3.5. Типы зарядки и коннекторы
- Сравнительный блок: **AC (медленная/Type 2, GB/T AC)** vs **DC (быстрая/GB/T DC, CCS2)** — карточки или таблица: мощность, ориентировочное время зарядки, для каких сценариев.
- Конкретные типы коннекторов и мощности сверить с данными в репозитории/CMS/публичной оферте; если данных нет — дать типовые для рынка Узбекистана (GB/T как основной стандарт) с пометкой `TODO: подтвердить спецификации у заказчика`.

### 3.6. «Как зарядиться» — пошаговая инструкция
- 3–4 шага в горизонтальный степпер (на мобильном — вертикальный): найти станцию → подключить коннектор → оплатить/запустить сессию (приложение/QR — уточнить по репо, есть ли моб. приложение) → отслеживать зарядку.
- Стилистически перекликается с таймлайном на главной (нумерация, линия-соединитель).

### 3.7. Тарифы (условная секция)
- Если тарифы есть в CMS или публичной оферте — вывести простую таблицу/карточки (AC/DC, сум за кВт·ч).
- Если нет — секцию не выдумывать; вместо неё CTA «Ознакомьтесь с условиями обслуживания» со ссылкой на публичную оферту (`/[locale]/chargingstation/public-offer`) — эта ссылка обязана сохраниться в любом случае.

### 3.8. FAQ
- Аккордеон на 5–6 вопросов: какие авто поддерживаются, как оплатить, что делать при сбое, работают ли станции круглосуточно, планы расширения сети, контакт поддержки.
- Использовать существующий компонент аккордеона, если найдётся; иначе — простой доступный `<details>`-based или headless-компонент без новых зависимостей.
- Добавить JSON-LD `FAQPage` schema.

### 3.9. CTA-блок
- Полноширинный блок в фирменном зелёном: «Хотите разместить станцию на своей территории / стать партнёром?» → кнопки на `/[locale]/contacts` и телефон +998 55-514-88-44.
- Далее — стандартный общий блок «Контакты» и футер без изменений.

## 4. Технические требования

1. **Не менять** header, footer, глобальные layout'ы и другие страницы. Изменения — только страница chargingstation + новые компоненты в её scope (или в общей папке компонентов, если так принято в проекте).
2. Все новые тексты — в систему локализации проекта, сразу EN/RU/UZ. Стиль перевода сверять с существующими словарями. Никаких хардкод-строк в JSX.
3. Изображения — только через `next/image` с корректными `alt` (локализованными), `sizes` и приоритетом для hero.
4. Иконки — inline SVG или существующий подход проекта; без установки icon-библиотек, если их нет.
5. Адаптивность: мобильный (≥360px), планшет, десктоп. Проверить сетки карточек на всех брейкпоинтах проекта.
6. Доступность: семантические заголовки (один `h1`), контраст по WCAG AA, фокус-стили, aria для аккордеона.
7. SEO: сохранить/улучшить metadata (title, description, OG через существующий `/api/og-image`), canonical, hreflang-паттерн как на других страницах. URL страницы не менять.
8. Производительность: без новых тяжёлых зависимостей; статическая генерация/ISR — как принято в проекте для контентных страниц.
9. Если контент страницы сейчас приходит из Hygraph — согласовать: либо новые поля в CMS (описать требуемую модель в `REDESIGN_NOTES.md`), либо временно локальные словари с планом миграции. Ничего в CMS без подтверждения не менять.

## 5. Что запрещено

- Выдумывать фактические данные (число станций, тарифы, мощности конкретных станций). Неизвестное — placeholder + `TODO`-комментарий + пункт в `REDESIGN_NOTES.md`.
- Ломать/удалять ссылку на публичную оферту.
- Вводить новые цвета, шрифты, дизайн-язык, конфликтующий с сайтом.
- Добавлять зависимости без явной необходимости.
